1. Abstract

Collecting Bottles on Web (ColB) is a handy tool constructed using web technologies. Created due to the lack of a platform for those with a passion for containers (yes, oddly specific, we know). Due to large range of content that can be provided by its users, it is an enjoyable tool to have.

2. Introduction

Collecting Bottles on Web is a tool that we have created as our project for Web Technologies, the capabilities of the website are the following :

  • User account creation and login
  • Search for items & filter posts based on tags with checkboxes
  • Each user has their own inventory table with categories, sub-categories and items
  • A logged-in user can create posts with their containers and their desired details
  • A user can select one (maybe more items) to share by the click of a button
  • An admin can connect to their account and they have the same features as a normal user, aside of some extra ones :
    • They're able to export stats via CSV and PDF
    • They're able to export a ranking of the most popular containers as a RSS data flux

3. Motivation

We chose this project to challenge ourselves, to discover what it takes to make such a specific tool considering the absence of such a concept. A application so similar to the concepts of the well known social media and classified advertisement platforms and yet so different.

4. Project functionalities

4.1 Search

This is the Search Page where we put an Input for the user to search for the desired items and a small menu where are shown some random itmes in case that user get's interested in any. This is the Results page, where user can see the results of his search and sort them in case that he is looking for the cheapest or most expensive items, it also has section with some other random selected items.The results are at maximum 4 on the same page, and user can also travel through pages to find the desired item. The results also have some basic information about the items.

This is the Product page, where users can see a more detailed information about the items they search for and also find some information about the Seller. It also has some more random options on the right side. In case that Link was modified it redirects user to page 404, where he is told that the item with the ID he is searching for does not exist.

4.5 Index page

This is the start page, a simple webpage in which the users can choose to go to one of the given pages, which has been designed and animated using this css. However, there is a small trick to this simple welcome, on access we also compile this php file, which signals the host to check for any missing tables within its database, if there are any missing they will automatically be created, thus making the webapp very easy to change host machines.

Aside of this webpage, all webpages that need any access to the database - include / import a simple php file, that does just that (creates a temporary connection).

4.6 Menu shortcuts

The menu and profile shortcuts are on most pages, designed via this css in which every button is customized, each icon is set towards a specific size, each button having it's own desired parameters. The animation of the other sub-buttons are done using this js, however the each main button has it's own twin. Once you click on a main button, it executes the animation and also sneakily changes its place with its twin (via a z-index), the twin does a reverse animation on click and once again changes place.

4.7 Sign-Up

The sign-up page is mostly made of the sign up form, its elements (from the text boxes and icons, to the writing and slide bar) are designed via this css. The emails and passwords are evaluated in real time, the passwords are hidden or shown (via *** characters) at the press of a button configured using this js.

Aside of the elements seen in the given pictures, there are a lot of things going on in the background. Due to some complications with sending the form data, all the checks (matching passwords / emails), non-empty fields and fields containing character specifics (such as having a @ followed by .com, or having a minimum of x characters - while having at least 1 special, 1 digit, etc), are all checked in the same javascript file using several functions and regexes, alerting the user of whatever issues there are (all the issues that are within the case at once, so the user doesn't need to guess).

Once the form details are acceptable they are also parsed using this php, where the username and email are first checked to see if there are any identical ones present in the database, if so the user is alerted and they can reintroduce once more the details. If not, then the user is welcomed with a successful account creation and redirected to the login page.

4.8 Login


Similarly to the sign-up page, its elements are designed via this css and a much simpler javascript file is used to hide and unhide the password, a difference being the addition of a few buttons.


However, unlike it's javascript file, the php file is far more complex. Here once again there are checks made with the database and a few conveniences are in place (for example the "username" to login can either be the created username or email). But there is an issue in comparison to the signup, yes we do send a successful login message and redirect the user, however how do we know how to access automatically further user data without constantly asking for the login details?

Well we implement a cookie that remembers the user's id, the cookie is removed after 24h, or it is modified / renewed through a same / different login. Once a user wants to connect to a different account we check for a existing cookie, if there is one, we give it the time of 0, afterwards we remember the new cookie with the respective id, thus making everything seemless for the user.

4.9 Forgot User / Pass


These 2 twin webpages are pretty self-explanatory, designed via this css and this css, so simple and yet there is much more behind the scenes.

On submission each webpage compiles a php file that helps recover the user or the password. Once the email is entered the data is sent to the respective php file, which then checks for any username / password associated to the email within the database, if there is none, then the user will be prompted a invalid email address message. However, if the email is valid, then the host automatically sends a email to the given address containing the requested information, afterwards redirecting the user to the login page.

The host has its own email address dedicated to such issues configured a smtp, for example if you are using Xampp you must configure the "php.ini" and "sendmail.ini" files. If you want to use a Google account for this then you have to secure and configure your Google account, as well as using a one time password (this can no longer be done easily through a simple option of allowing 3rd party apps to your account - feature removed on the 20th of May 2022).

4.10 Profile


Once the user logs in they are redirected to this webpage, where they can insert whatever profile details they'd like, these details afterwards being used in their posts and inventory, thus reducing the chances of spam / fraudulent posts.

When the user confirms the given details they will be automatically update the table below, showing the user a preview of the data, if they dislike it or a mistake was made, they can enter and submit new details.


As usual, the webpage is designed via a css, however this time, since we need real time data to show, the webpage and the information requests are implemented at once in one php, the submission of the introduced information are done via a different php.

4.11 About Us


This is a more informative webpage, as usual, designed using a css. This webpage has also a few perks, any user logged in or not can check these details and send a contact form, the difference in between logged in and non-logged in users being that for logged in users the Name and Email are automatically extracted from the database.

Similarly to the forgotten details pages, it uses a smtp to send the details, the information being parsed via a php file where the given information is rewritten in a different format so that the ColB team can easily understand from whom the email is and what is that they want to communicate.

5. How the team worked

For the management of the project and each of it's parts we used Github, updating our branch with whatever we worked on and when we felt like we have improved the project altogether, the team leader would put everything in one place, (if necessary) make some file location error handling and then push it to main.

All members contributed to the making of this project, each of us working at our own pace and doing commits when we felt proper, in our own branch. The detailed progress can be seen in the commits section, the main branch being used to update the project as a whole.
